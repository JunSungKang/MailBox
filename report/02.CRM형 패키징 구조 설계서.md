# 02. CRM형 패키징 구조 설계서

작성일: 2026-05-02

## 1. 목적

메일 서버 프로젝트를 `/home/jskang/crm-project`의 최종 패키징 방식과 같은 기준으로 정리한다.

핵심 원칙은 다음과 같다.

- 제품 소스와 납품 패키지를 분리한다.
- 고객사 납품 패키지는 컨테이너 이미지 pull 방식으로 실행한다.
- 고객사 서버에는 Node.js, npm 같은 빌드 도구 설치를 요구하지 않는다.
- 실제 운영 `.env`, 로그, 메일 데이터, 빌드 캐시는 납품 패키지에 포함하지 않는다.
- 릴리즈 단위는 compose, env 예시, 이미지 lock, 문서, 점검/백업 도구로 구성한다.

```text

│   ├── .env.example
```

CRM의 `app-server/db-server/db-tools` 구조는 메일 서버 도메인에 맞게 `mail-server/mail-tools`로 대응한다.

## 4. 이미지 명명 규칙

```text
ghcr.io/junsungkang/mail-server-smtp-receiver:v1.0.0
ghcr.io/junsungkang/mail-server-imap-mailbox:v1.0.0
ghcr.io/junsungkang/mail-server-webmail-ui:v1.0.0
```

이미지 digest는 `images.lock`에 기록한다.

## 5. 생성 절차

```bash
cd source
scripts/images/build-images.sh
scripts/images/push-images.sh
scripts/images/write-images-lock.sh
scripts/package/build-release.sh
```

내부 검증은 `source/deploy/local/docker-compose.yml`로 수행하고, 고객사 납품은 `source/deploy/single-node/docker-compose.yml` 기반 release 패키지로 수행한다.
